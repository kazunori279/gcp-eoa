The Crisis Briefing

Signal Failure at London St Pancras

Peak evening departures. A signal failure at the St Pancras throat has halted all cross-border services. Hundreds of travelers, one question repeated a thousand ways: "Will I still make it — and how?"

💬 Why a Chatbot Fails Here

A plain LLM knows the rulebook, but not whether your train is actually running right now. It will confidently recite timetable policy — then hallucinate a service that was cancelled an hour ago.

An agent is different: it has tools that can query the published schedule (what should happen), read the disruption feed (what is happening), and compute the join of those two — giving you a deterministic, verifiable answer: the delay, and a concrete reroute.

🎯 Your Deliverable

By the end of this workshop you will have built a governed, deployed, optimized agent that a non-technical ops manager can simply chat with in Gemini Enterprise:

user prompt"Trains out of St Pancras are disrupted — what do I tell travelers heading to Paris?"

And the agent will answer with a concrete, verified reroute — not a guess.

The Data — Static GTFS + Simulated Disruption

No Live APIs

We use static data only. This keeps the lab 100% reliable in a room of 100+ people on conference Wi-Fi, and keeps the focus on the platform, not on flaky feeds.

📅 The GTFS Bundle

A static London GTFS dataset covering Eurostar / St Pancras cross-border rail:

⚠️ The Disruption File

A small file that simulates the "signal failure at St Pancras" crisis. You'll layer this on top of the GTFS schedule data to create the disruption scenario.

The disruption specifies:

  • Which trip IDs are affected (cancelled / delayed)
  • The delay magnitude (in minutes)
  • Which stops are impacted
  • The cause ("signal failure") and time window

Your agent's tools must reconcile this file against the GTFS schedule to produce answers.

🔧 Tools Specification

The three tools the agent needs, with expected behavior.

Agent Context · Tools ## Agent Tools (3 required) ### Tool 1: get_scheduled_departures Purpose: Look up "what SHOULD be running" from the static GTFS data. Inputs: - station_name: str — name or ID of the station (e.g., "St Pancras") - time_window_start: str — ISO time or HH:MM - time_window_end: str — ISO time or HH:MM (optional, default +2h) Output: List of scheduled departures with trip_id, route, destination, departure_time, and stop_sequence. Data source: GTFS files (stops.txt, stop_times.txt, trips.txt, routes.txt, calendar.txt) Key join: stop_times.trip_id → trips.trip_id → routes.route_id ### Tool 2: check_disruptions Purpose: Read "what IS happening" from the disruption feed. Inputs: - station_name: str (optional — filter by affected station) - trip_id: str (optional — check a specific service) Output: List of disruptions with trip_id, affected_stops, delay_minutes, status (cancelled | delayed), cause, and time_window. Data source: disruptions.json ### Tool 3: compute_reroute Purpose: Given an origin, destination, and the current disruptions, find an alternative path across the 17-station network. Inputs: - origin: str — starting station - destination: str — target station - departure_after: str — earliest acceptable departure time Output: Recommended route as a list of legs, each with departure_time, arrival_time, station_from, station_to, trip_id, and route_name. Total journey time and number of changes. Logic: Filter out disrupted services, search remaining connections for a valid path (BFS/shortest-path over the station graph). ### Reconciliation pattern (critical) The agent MUST call Tool 1 + Tool 2 before answering any disruption question. The join of schedule × disruption on trip_id/stop_id is the core pattern that makes this agent useful. Tool 3 is used when a reroute is needed.

📝 System Instruction Template

Suggested system prompt for the transit agent.

Agent Context · System Instruction ## System Instruction for the Transit-Crisis Agent You are a transit assistant for cross-border rail services departing from London St Pancras. A signal failure is disrupting services. Rules: 1. ALWAYS reconcile the published schedule against the disruption feed before answering. Call get_scheduled_departures AND check_disruptions for every query — never answer from memory alone. 2. NEVER invent a train service. Only reference trips that exist in the GTFS data. If no data is available, say so. 3. When a service is disrupted, proactively call compute_reroute to offer a concrete alternative. Include departure time, route, and number of changes. 4. Stay calm and factual. Travelers are stressed — be concise, clear, and reassuring without overpromising. 5. End every response with ONE clear recommended action (e.g., "Take the 14:05 via Brussels Midi, arriving Paris 17:32"). 6. If the disruption feed is unavailable or returns an error, tell the user you cannot confirm live status and recommend they check departure boards.
