Teaching Beat: The Reconciliation Pattern

The magic is the join of schedule × disruption on trip_id / stop_id. This is the pattern that makes the agent more than a chatbot. Without it, the agent knows timetables but not reality.

The 3 Tools — In Detail

Your agent needs three deterministic tools. Each one answers a specific question. Together, they implement the reconciliation pattern.

1 get_scheduled_departures

Purpose: Look up "what SHOULD be running" from the static GTFS data.

get_scheduled_departures(station_name: str, time_window_start: str, time_window_end: str = None) → list[dict]
ParameterTypeDescription
station_namestrName or ID of the station (e.g., "St Pancras")
time_window_startstrStart time (HH:MM or ISO)
time_window_endstrEnd time (optional, default +2h)

Returns: List of departures with trip_id, route_name, destination, departure_time, stop_sequence.

Data source: stops.txt → stop_times.txt → trips.txt → routes.txt (joined on stop_id, trip_id, route_id).

2 check_disruptions

Purpose: Read "what IS happening" from the disruption feed.

check_disruptions(station_name: str = None, trip_id: str = None) → list[dict]
ParameterTypeDescription
station_namestrFilter by affected station (optional)
trip_idstrCheck a specific service (optional)

Returns: List of disruptions with trip_id, affected_stops, delay_minutes, status (cancelled/delayed), cause, time_window.

Data source: data/disruptions.json

3 compute_reroute

Purpose: Find an alternative path avoiding disrupted services.

compute_reroute(origin: str, destination: str, departure_after: str) → dict
ParameterTypeDescription
originstrStarting station
destinationstrTarget station
departure_afterstrEarliest acceptable departure

Returns: Recommended route as a list of legs (each with departure_time, arrival_time, station_from, station_to, trip_id, route_name), plus total_journey_time and num_changes.

Logic: Filter out disrupted services, search remaining connections for a valid path (BFS / shortest-path over the 17-station graph).

Test & Trace — Running the Agent Locally

🎮 Launch the Playground

shelladk web

This starts the ADK local development playground — a web UI where you can chat with your agent, see tool calls in real time, and inspect the reasoning trace.

🧪 The Test Query

playground"My 13:31 St Pancras–Paris service is affected — how do I still get to Paris?"

🔍 What to Look For in the Trace

Open the reasoning trace after the agent responds. You should see:

OrderTool calledExpected behavior
1 get_scheduled_departures Queries St Pancras departures around 13:31 — finds the Paris service
2 check_disruptions Checks the 13:31 trip — finds it's cancelled/delayed due to signal failure
3 compute_reroute Finds alternative: e.g., via Lille or Brussels, with a later departure

✅ What a Good Answer Looks Like

The agent should:

  • Confirm the 13:31 service is affected (with specific status: cancelled/delayed)
  • Explain why (signal failure at St Pancras)
  • Offer a concrete reroute with specific times and stations
  • End with one clear recommended action
Example good answer

"Your 13:31 St Pancras–Paris Eurostar is cancelled due to the signal failure. The next available route is the 14:05 via Brussels Midi, changing in Brussels with arrival at Paris Gare du Nord at 17:32 (1 change, ~3h27m total). Recommended: head to Platform 5 for the 14:05 Brussels service."

🚨 Common Issues

SymptomLikely causeFix
Agent doesn't call tools Missing/bad docstrings or type hints Check tool function signatures and docstrings
Agent invents train times System instruction not enforced Strengthen: "NEVER invent — only reference trips in the data"
Tool returns empty results GTFS data path wrong or not loaded Verify data/gtfs/ files exist and paths in tools are correct
Reroute tool fails Graph not built from GTFS connections Ensure the tool builds a station graph from stop_times.txt

🔧 Tool Specifications

Detailed specs for the 3 tools the coding agent should implement.

Agent Context · Tool Specs ## Tool Specifications for tools.py All tools should use ADK's FunctionTool pattern. Write clear docstrings and type hints — the model uses these to decide when/how to call tools. ### Tool 1: get_scheduled_departures Purpose: Query "what SHOULD be running" from static GTFS data. Signature: def get_scheduled_departures( station_name: str, time_window_start: str, time_window_end: str = None ) -> list[dict]: Behavior: 1. Look up station in data/gtfs/stops.txt by stop_name 2. Find departures in data/gtfs/stop_times.txt matching stop_id within the time window 3. Join with data/gtfs/trips.txt on trip_id for route info 4. Join with data/gtfs/routes.txt on route_id for route name 5. Return list of dicts: {trip_id, route_name, destination, departure_time, stop_sequence} Key joins: stops.stop_id → stop_times.stop_id, stop_times.trip_id → trips.trip_id, trips.route_id → routes.route_id ### Tool 2: check_disruptions Purpose: Read "what IS happening" from the disruption feed. Signature: def check_disruptions( station_name: str = None, trip_id: str = None ) -> list[dict]: Behavior: 1. Read data/disruptions.json 2. Filter by station_name or trip_id if provided 3. Return list of dicts: {trip_id, affected_stops, delay_minutes, status ("cancelled"|"delayed"), cause, time_window} ### Tool 3: compute_reroute Purpose: Find alternative route avoiding disrupted services. Signature: def compute_reroute( origin: str, destination: str, departure_after: str ) -> dict: Behavior: 1. Build a station graph from data/gtfs/stop_times.txt connections 2. Get current disruptions (call check_disruptions internally or read disruptions.json) 3. Remove disrupted trips from available connections 4. BFS/shortest-path from origin to destination 5. Return: {route: [{departure_time, arrival_time, station_from, station_to, trip_id, route_name}], total_journey_time, num_changes} ### Critical: The Reconciliation Pattern The agent MUST call Tool 1 + Tool 2 before answering any disruption question. The join of schedule × disruption on trip_id/stop_id is the core pattern. Tool 3 is used when a reroute is needed. The system instruction must enforce this sequence.

📝 System Instruction

The system instruction to write for the agent.

Agent Context · System Instruction ## System instruction for agent.py You are a transit assistant for cross-border rail services departing from London St Pancras. A signal failure is disrupting services. Rules: 1. ALWAYS reconcile the published schedule against the disruption feed before answering. Call get_scheduled_departures AND check_disruptions for every query — never answer from memory alone. 2. NEVER invent a train service. Only reference trips that exist in the GTFS data. If no data is available, say so. 3. When a service is disrupted, proactively call compute_reroute to offer a concrete alternative. Include departure time, route, and number of changes. 4. Stay calm and factual. Travelers are stressed — be concise, clear, and reassuring without overpromising. 5. End every response with ONE clear recommended action (e.g., "Take the 14:05 via Brussels Midi, arriving Paris 17:32"). 6. If the disruption feed is unavailable or returns an error, tell the user you cannot confirm live status and recommend they check departure boards.
