### Scenario tests against the DEPLOYED agent
Every attempt to invoke the deployed engine (set/recall home station; average-delay math)
returned an error instead of an answer — see root causes. (Locally, in M1, the same agent
answered all three correctly.)

## Root causes found (a 3-layer stack)

1. **Deploy-time init crash (Step 3, self-fixed).** `memory_service_builder` raised when
   `GOOGLE_CLOUD_AGENT_ENGINE_ID` was unset during container init; it didn't fall back to
   `InMemoryMemoryService` robustly. agy hardened the fallback and redeployed — fixed.
2. **`Invalid Session resource name` (the big one).** `app/agent.py` runs
   `os.environ["GOOGLE_CLOUD_LOCATION"]="global"` at import. `VertexAiSessionService` then
   builds `…/locations/global/…/sessions/…`, but the **Vertex AI Sessions API is regional**
   (us-east1) → backend 400 → every deployed call dies in session lookup → empty response.
   Traces to `gcp.location=global` set during M0 `agents-cli setup`. Partial fix (pin
   session/memory services to us-east1) reduced but did not fully clear it — the genai
   client/ADK still resolve location from the env var in places.
3. **`Event loop is closed` in `async_stream_query`.** A cached `api_client` bound to a
   completed event loop is reused across the container's request threads. agy's fix
   (clear the cached client per request) was in progress when stopped.

> Note: the **session_id _normalization_** the workshop emphasizes in M2 Step 1 (the
> GE-session-id fix) addresses a *different* problem (full-path session ids from Gemini
> Enterprise) and did **not** prevent any of the above. It can give false confidence that
> deployed sessions will work.

## Process finding (the key one) — no pre-deploy local test

Every failure above is in the **runtime wrapper + real Sessions API path**, which the
workshop never exercises before deploying ("verify session state locally" only tests the
plain local agent, not `agent_runtime_app`'s `async_stream_query` against
`VertexAiSessionService`). So each bug cost a **~10-min deploy** to discover something a
**~10-sec local test** would catch. agy fell into a blind fix→deploy→test loop (~8 deploys).
A `scripts/runtime_smoke.py` that drives `async_stream_query` + `VertexAiSessionService`
@us-east1 in one asyncio loop reproduces all three errors locally. (This redirect was
introduced at the end of the run but not completed.)

## Improvement plan for `m2.html`

- **P1 — Don't set `GOOGLE_CLOUD_LOCATION="global"`.** This single choice (inherited from
  M0's `gcp.location=global`) breaks the regional Sessions API and is the primary reason the
  deployed agent returns empty. Either set the project/agent location to the deploy region
  (`us-east1`) from M0, or have M2 explicitly pin the session/memory service region. **This
  is the most important fix; without it the deployed agent is non-functional, which also
  breaks M5.**
- **P2 — Add a pre-deploy local smoke-test step** that exercises the *runtime wrapper*
  (`async_stream_query` + `VertexAiSessionService`@region) before any `agents-cli deploy`.
  Converts ~10-min discovery cycles into seconds and is the single biggest time saver.
- **P3 — Ship a robust `agent_runtime_app.py` template** (or have `scaffold enhance`
  generate one) that already: falls back to `InMemoryMemoryService` when no engine id;
  resolves session/memory region correctly; and avoids the stale-`api_client`/event-loop
  reuse. Today every student would hit the same 3 bugs.
- **P4 — Recalibrate the session_id-normalization framing.** Make clear it solves GE's
  full-path session ids, *not* deployed-session validity — so students don't assume "deploy
  will just work."
- **P5 — Project-name drift** (carried from M1): M2 Step 1's retry block hardcodes
  `cd ~/transit-assistant`, but the project is `transit-crisis-agent`. Pin the name or
  derive it.
- **P6 — Budget/expectation:** as written, M2's deploy+debug is far longer than the stated
  25 min once these bugs bite. Fixing P1–P3 should bring it back in range.

## Recommended restructure: "code first, test local, deploy once"

The single biggest duration win is to **collapse M2's two planned deploys (Step 1 initial +
Step 3 redeploy) into one**, and move all the failure-prone runtime work *before* it behind
a local test. This combines P1+P2+P3 into a new step order:

| Step | What | Deploy? |
|---|---|---|
| 1 — Prep runtime entrypoint | `scaffold enhance` the Agent Runtime app; bake in session_id normalization **+ regional (not `global`) session/memory builders + per-request client**. No deploy. | — |
| 2 — Sessions + Memory Bank | code only; verify locally with `InMemory*` services | — |
| 3 — Code execution | code only; local `agents-cli run` math test | — |
| 4 — Local runtime smoke test | `scripts/runtime_smoke.py` drives the real deployed entry path locally; iterate in seconds until green | — |
| 5 — Deploy once | one `agents-cli deploy --agent-identity` shipping sessions + memory + code-exec | ✅ ×1 |
| 6 — Test deployed (playground) | confirm it answers | — |

**Net: 2 deploys → 1**, and the ~45-min debug loop disappears because the blockers are
caught locally.

**Key feasibility point — the smoke test does NOT need a live engine.** Vertex AI sessions
live *under* the reasoningEngine (`…/reasoningEngines/{id}/sessions/…`), so the real
sessions API can't be hit before the first deploy. But all three M2 bugs are catchable
locally without an engine:
- **init crash** → `import app.agent_runtime_app` + build the app;
- **`global` location** → assert the session/memory builders resolve region to the deploy
  region (config check, no API call);
- **`Event loop is closed`** → drive `async_stream_query` with `InMemorySessionService`
  through one asyncio loop — the loop/client-reuse bug reproduces with no engine.

**Trade-off:** the current flow deploys early so students see it live before adding
features; deploy-once defers that to the end. Given the 25-min budget and ~10-min deploy,
the time win is worth it — and Step 3 already says "single redeploy … to save time," so this
finishes that intent.
