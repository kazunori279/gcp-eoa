## What worked well

- **Self-verification:** in Step 2 the agent wrote and ran its own test harness for the
  tools before declaring done, and ran lint/type-checks in Steps 2‚Äì3. High-quality loop.
- The system instruction's "always reconcile" rule actually manifested ‚Äî every scenario
  called both schedule and disruption tools before answering.
- `gemini-3.5-flash` was kept as specified (the prompt's "do not change it" guard worked).

## Issues / friction observed

1. **Project name drift (will break M2).** Step 1 doesn't pin a name, so agy chose
   `transit-crisis-agent`. But **M2 Step 1's retry block hardcodes `cd ~/transit-assistant`**
   ‚Äî that path won't exist. **High priority.**
2. **Tools live in `app/agent.py`, not `tools.py`.** M2 Step 1 says "update the data-path
   resolution in the code (e.g. `tools.py`)". The "e.g." softens it, but an agent looking
   for `tools.py` could be confused.
3. **Playground is browser-only.** Step 4 ("share the link for me to test") assumes a human
   with a browser; fine for the live workshop, but there's no headless validation path. The
   agent also fumbled invocation (`--help`, a probe) before finding `agents-cli run --url ‚Ä¶`.

## Improvement plan for `m1.html`

- **P1 ‚Äî Pin the project name** in Step 1 ("scaffold it as `transit-assistant`") OR change
  M2 Step 1's retry block to not hardcode the directory (derive it / `cd` into the scaffolded
  project). This is the one change that prevents a hard M2 failure.
- **P2 ‚Äî Make the tools-file reference robust.** Either instruct Step 2 to put tools in
  `tools.py`, or change M2's wording to "wherever the tools read the data files" instead of
  naming `tools.py`.
- **P3 ‚Äî Add an optional headless test recipe** to Step 4: the exact
  `agents-cli run --url http://127.0.0.1:8080 --mode adk "<query>"` command, so students (or
  an agent) can validate without the browser and without trial-and-error.
