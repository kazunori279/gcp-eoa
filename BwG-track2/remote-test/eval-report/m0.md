## What worked well

- The `plan.md`-after-every-step discipline held: the agent re-read and edited `plan.md`
  at steps 3 & 4 without being re-told. Good "don't run ahead" design.
- The empty-`mcp_config.json` bug is handled robustly — the agent wrote a defensive Python
  merge rather than blindly overwriting.
- Native DK MCP grounding works in this lab, so the workshop's "happy path" is real.

## Improvement plan for `m0.html`

**Priority 1 — shrink the Step 1 prompt by fixing the lab image.** `ctx-dk-mcp` is the
most complex prompt in the whole workshop (~1.6 KB) almost entirely because it works around
a *lab environment defect*: `~/.gemini/config/mcp_config.json` ships as a 0-byte file → JSON
parse failure → all MCP tools silently disabled. If the lab image shipped `{}` (or a
pre-seeded config), this prompt could drop to ~3 lines ("enable the API, add this server,
restart"). **Recommendation:** file a lab-image fix; keep the defensive version only as a
fallback callout.

**Priority 2 — collapse the data download to one self-healing prompt.** Steps currently
ship `cmd-download-data` *and* `cmd-download-data-fallback` as two separate copy-paste
blocks. Merge into one prompt: "try the Eurostar URL; if it fails, use the GitHub mirror."
Removes a manual decision from the student.

**Priority 3 — reconsider the `dk.py` fallback's prominence.** Native `answer_query` worked
cleanly. The `ctx-dk-helper`/`dk.py` path now reads as the "expected" escape hatch; demote
it to a small "if blocked" note so students don't pre-emptively use the fallback (which
would skip the real MCP experience the later modules depend on).

**Priority 4 — note the ADC identity explicitly.** Step 3 authenticates as
`workstation-sa@…`, not the student. Later modules (M2 deploy, M3 IAM) reason about
identity; a one-line callout in M0 that "the cockpit uses the workstation service account
via ADC" pre-empts confusion.
