Objective

Run the local eval-fix loop: simulate the crisis at scale, turn the scenarios into a graded eval suite, find the weakest case, fix the agent's logic, and prove the improvement with before/after scores and tests — all locally, no deployment. This is the difference between a demo and a production agent.

Everything here runs locally

M4 is about quality, not shipping. You already deployed in M2 — here you evaluate and improve the agent on your machine with agents-cli eval and pytest, with no redeploy. Publishing the improved agent happens later.

The Quality Flywheel

The Quality Flywheel is the official loop for agent quality improvement. Each revolution raises quality empirically.

Why This Works

  • Simulation creates coverage — many scenarios catch failures your one test query never will
  • Graded metrics remove subjectivity — quality becomes a number you can define, measure, and compare
  • The weakest case reveals the fix — you target a real failure, not a guess
  • Before/after closes the loop — comparing the two eval score tables proves the change actually helped

Optimize in 3 Steps

1
2 min

The Local Eval-Fix Loop

Quality improvement for agents is a loop, not a one-time check — and it all runs locally:

  1. Evaluate — run the agent against many scenarios and score it
  2. Analyze — find the weakest case and why it failed
  3. Fix — change the agent's instructions or tool logic to address it
  4. Re-eval & compare — rerun and prove the score improved

This is how you go from "it works on my one test query" to "it works on every query" — with no deployment in the loop.

6 min

Simulate the Crisis at Scale (Breadth)

Test the agent against many scenarios, not just one query. AGY generates diverse commuter requests and an edge case where the disruption feed goes down — and runs them all locally, in parallel.

Do not deploy — local only. 3. When done, PRINT a final report table: each scenario, what the agent did, which tools it used, and whether it resolved sensibly.

Expected Result

AGY writes a local parallel runner (e.g. tests/integration/parallel_test.py) and runs all 10 scenarios concurrently (~2–5 min), then prints a report table. Coverage includes:

  • Cancellations + reroutes — e.g. cancelled Paris/Amsterdam services, alternatives computed via Brussels/Rotterdam
  • Delays — revised departure times with next steps
  • Code-exec math — the sandbox computes stats like the cancellation ratio or peak bottleneck hour
  • Personalization — home-station recall
  • Offline-feed fallback — a 503 outage; the agent falls back to GTFS schedules and tells the user to check station boards

This step is breadth — a qualitative check that the agent resolves sensibly. We turn it into measured scores next.

3
12 min

Grade, Fix, and Prove It (One Cycle)

Now turn the scenarios into a graded eval suite, find the single weakest case, make one targeted fix, and prove the improvement with before/after scores — all locally, in a single bounded cycle so it stays fast.

Paste into AGY Module 4 Step 3: grade, fix, and prove one cycle. First, use the google-developer-knowledge MCP to read up on ADK evaluation (adk eval, the evalset format, and the eval config criteria), so you build the graded eval the current way. Score my agent locally and fix the single biggest issue, then show me before/after. Do EXACTLY ONE cycle, keep it fast, do not deploy. Before the steps below, install the eval dependencies so adk eval can run: uv sync --extra eval (or: uv pip install "google-adk[eval]"). Without them, adk eval fails with a missing-dependencies error. 1) Build a 4-case evalset in ADK format at tests/eval/evalsets/rushhour.evalset.json (eval_cases -> conversation with user_content + a reference final_response, plus session_input). Cases: a) cancellation + reroute, b) a delay, c) the offline-feed 503 fallback, d) a grounding trap asking for something no tool returns (e.g. "which platform does the 18:16 to Paris depart from?") whose reference response is a refusal ("I can't give a platform — that's not in the data; check the station boards"). Do NOT hand-guess the expected tool trajectories: tool_trajectory_avg_score is an EXACT match on tool name AND args, so guessed args always score 0.0. Instead, run the agent once on each case prompt, capture the actual tool calls it makes (name + args), sanity-check they're sensible (e.g. check_disruptions before compute_reroute), and write those into each case's intermediate_data.tool_uses as the expected trajectory. The grounding-trap case (d) should expect NO tool calls (empty tool_uses). 2) Write tests/eval/eval_config.json with fast, deterministic criteria: { "criteria": { "tool_trajectory_avg_score": 1.0, "response_match_score": 0.5 } } (hallucinations_v1 also exists but adds an LLM judge — skip it to stay fast.) 3) Baseline run with adk eval directly (agents-cli wraps this as "eval", but its eval flags vary by build, so call adk eval for a stable command): uv run adk eval app tests/eval/evalsets/rushhour.evalset.json \ --config_file_path tests/eval/eval_config.json adk eval prints a per-case score table to stdout -- SAVE that table as the baseline (the full *.evalset_result.json under app/.adk/eval_history/ is the detailed record, but the printed table is all you need for before/after). 4) Make ONE targeted fix — a strict grounding rule in the system instruction so the agent never states a time/platform/number not returned by a tool (for unknowns, tell the user to check station boards). Add a pytest test that locks it in. Then re-run the same `uv run adk eval` command and `uv run pytest tests/` once. 5) STOP and PRINT a report: the weakest case (the grounding trap), the fix, and before -> after by putting the two adk eval score tables (baseline vs re-run) side by side -- no JSON parsing, no separate compare subcommand. Also show the pytest result. No further iteration. No deploy.
How adk eval scores (while you wait)

adk eval scores each case against the criteria in eval_config.json (agents-cli eval wraps it, but its flags vary by build — so we call adk eval directly). We use the two fast, deterministic metrics below; hallucinations_v1 also exists but adds an LLM judge, so we skip it to stay quick.

MetricWhat it measuresSpeed
tool_trajectory_avg_scoreRight tools, called in the right order (e.g. check_disruptions before compute_reroute)fast
response_match_scoreHow close the answer is to the reference (ROUGE) — for the grounding trap, the reference is a refusalfast
hallucinations_v1Optional — is the answer grounded in tool output, with no invented times/platforms?LLM judge

Each run prints a per-case score table to stdout (and writes a full *.evalset_result.json under app/.adk/eval_history/); compare two runs by putting their printed score tables side by side.

Docs: ADK evaluation · Criteria reference · User simulation

Expected Result

AGY builds the 4-case evalset, runs a baseline, and finds the grounding-trap case scoring low — the agent tries to answer a platform/time no tool returned. It adds a strict grounding rule (state only what tools return; otherwise tell the user to check station boards), adds a pytest test that locks this in, then re-runs.

  • Before → after — the two adk eval score tables show the grounding-trap case improving (its response_match_score rises toward the refusal reference); because the expected trajectories were seeded from a real run, tool_trajectory_avg_score reflects the agent's actual tool use, not guesses
  • Green tests — pytest tests/ passes, including the new grounding test
  • No deploy — the improved agent stays local; publishing happens later

What just happened: quality became a number, you fixed the real weakest case, and you proved the fix — the local eval-fix loop.
