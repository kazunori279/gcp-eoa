Agent Context Test Plan

Validate that workshop context snippets are sufficient for a coding agent to build the transit-crisis ADK agent end-to-end
What are we testing?
Whether a coding agent (simulating Antigravity) can build a complete, syntactically valid ADK transit-crisis agent using only the agent context snippets that workshop participants copy-paste from the HTML pages.

Methodology

Launch an independent subagent with the exact text from 4 context blocks — the same content participants will paste into Antigravity. The subagent:

  1. Generates synthetic GTFS data — 17 stations, ~20 trips, 5 routes, plus disruptions.json
  2. Scaffolds the ADK project — agent/ directory with __init__.py, agent.py, tools.py, and pyproject.toml
  3. Implements 3 tools — get_scheduled_departures, check_disruptions, compute_reroute
  4. Writes the system instruction — reconciliation rules per the context
  5. Self-validates — Python AST parse to confirm no syntax errors

What this reveals

  • Missing info — Where the subagent had to guess or make assumptions not covered by the context blocks
  • Ambiguities — Where the subagent made reasonable but potentially wrong choices
  • Structural issues — Where generated code doesn't follow the expected project layout

Each gap becomes a TODO for improving the workshop HTML before the event.

Constraint
The subagent does NOT install dependencies or run the agent. It generates code and validates syntax only. The test proves instructional sufficiency, not runtime correctness.

Fixes Applied

Fix 1: m0.html — Remove file-path-laden individual blocks

Problem: Five individual context blocks (ctx-project, ctx-tools, ctx-sysinstruction, ctx-prompts, ctx-env) contained file paths like stops.txt, data/gtfs/, disruptions.json, and a full workspace tree. When pasted into AGY, the agent interpreted these as actionable and spent 20+ tool calls searching for the files.

Root cause: AGY treats any mention of file paths as actionable context. At the intro stage, no files exist yet — they arrive in M0.

Fix: Removed all 5 individual blocks (193 lines). The Agent Context tab now has a single "Mission Briefing" block with scenario background only — no file names, no paths, no build instructions.

Commit: be0710e

Fix 2: m0.html — Prevent premature building

Problem: The combined context block ended with a "What to build next (M1)" section containing numbered steps: "1. Scaffold ADK project, 2. Implement 3 tools, 3. Write system instruction, 4. Test." AGY interpreted this as "do these things now" and immediately started scaffolding, searching for data, and building — 40+ tool calls before timeout.

Root cause: Numbered action items without a "wait" signal read as imperatives to a coding agent.

Fix: Replaced the numbered build steps with a brief "Preview of Module 1" paragraph and added "Do NOT start building yet — this context is a workspace summary only. Wait for the Module 1 instructions."

Commit: 59bb63b

Key Learnings

Authoring agent context blocks for AGY

  1. File paths are actionable. Any mention of a file name (stops.txt, disruptions.json) triggers AGY to search for it. Only include file paths in blocks where the files already exist in the workspace.
  2. Numbered steps are imperatives. A list like "1. Scaffold, 2. Implement, 3. Test" is interpreted as "do this now." Use passive language ("In the next module you will…") for previews of future work.
  3. "Do NOT" guards work. Explicit "Do NOT start building yet" text at the end of a block reliably prevents AGY from taking action.
  4. AGY reads its own skills first. AGY consults built-in skills (adk-dev-guide, adk-cheatsheet, adk-scaffold) before acting on user context. This means context blocks don't need to specify ADK API details — AGY discovers them.
  5. AGY asks before destructive actions. M5 showed AGY verifying the agent locally, then asking for deploy approval rather than deploying immediately. This is ideal workshop behavior.
